# Feature: Phase 4 — Metrics Engine, Benchmarking & Telemetry

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Add a SQLite-backed metrics engine that passively records events from every PICE workflow command, implements `pice metrics` and `pice benchmark` as real commands (replacing Phase 1 stubs), and provides opt-in anonymous telemetry with full transparency. This turns PICE from a "fire and forget" orchestrator into a tool that proves structured AI coding works with data.

## User Story

As a developer using the PICE workflow, I want to see aggregate quality data (pass rates, score trends, loop counts) so that I can prove structured AI coding produces measurably better code than ad-hoc approaches.

## Success Criteria

- [ ] `pice evaluate` records evaluation results (scores, pass/fail, tier, models) to SQLite
- [ ] `pice plan`, `pice execute`, `pice commit` record lifecycle events to SQLite
- [ ] `pice metrics` displays aggregate data (total loops, pass rate, avg score, 30-day trend)
- [ ] `pice metrics --json` and `pice metrics --csv` produce valid machine-readable output
- [ ] `pice benchmark` shows before/after comparison using git stats + metrics data
- [ ] `pice init` creates a properly initialized SQLite database (not an empty file)
- [ ] `pice status` shows latest evaluation result per plan
- [ ] Telemetry is opt-in, anonymized, logged to `.pice/telemetry-log.jsonl`, and inspectable
- [ ] Telemetry endpoint failures are silent (debug-level only)
- [ ] All validation commands pass with zero errors
- [ ] No regressions in existing functionality

### Task 2: Create Metrics Database Module

**File:** `crates/pice-cli/src/metrics/db.rs`
**Action:** CREATE
**Description:** `MetricsDb` struct wrapping `rusqlite::Connection`. Open with WAL mode (`PRAGMA journal_mode=WAL`). Schema versioning via `schema_version` table (read current version, run migrations sequentially). V1 schema creates four tables:

```sql
-- Schema version tracking
CREATE TABLE IF NOT EXISTS schema_version (
    version INTEGER NOT NULL
);

-- Evaluation results
CREATE TABLE IF NOT EXISTS evaluations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    plan_path TEXT NOT NULL,
    feature_name TEXT NOT NULL,
    tier INTEGER NOT NULL,
    passed INTEGER NOT NULL,         -- 0 or 1
    primary_provider TEXT NOT NULL,
    primary_model TEXT NOT NULL,
    adversarial_provider TEXT,       -- NULL for Tier 1
    adversarial_model TEXT,          -- NULL for Tier 1
    summary TEXT,
    timestamp TEXT NOT NULL           -- ISO 8601 UTC
);

-- Per-criterion scores (FK to evaluations)
CREATE TABLE IF NOT EXISTS criteria_scores (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    evaluation_id INTEGER NOT NULL REFERENCES evaluations(id),
    name TEXT NOT NULL,
    score INTEGER NOT NULL,
    threshold INTEGER NOT NULL,
    passed INTEGER NOT NULL,
    findings TEXT
);

-- Lifecycle events (plan created, execute start/complete, commit)
CREATE TABLE IF NOT EXISTS loop_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    event_type TEXT NOT NULL,         -- 'plan_created', 'execute_started', 'execute_completed', 'commit'
    plan_path TEXT,                   -- NULL for events not tied to a plan
    timestamp TEXT NOT NULL,
    data_json TEXT                    -- Optional structured payload
);

-- Telemetry outbox (events queued for sending)
CREATE TABLE IF NOT EXISTS telemetry_queue (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    payload_json TEXT NOT NULL,
    created_at TEXT NOT NULL,
    sent INTEGER NOT NULL DEFAULT 0   -- 0 = pending, 1 = sent
);
```

Public API:
- `MetricsDb::open(path: &Path) -> Result<Self>` — open or create, run migrations
- `MetricsDb::open_in_memory() -> Result<Self>` — for testing
- `fn conn(&self) -> &Connection` — borrow the connection for store/aggregator

```rust
pub mod aggregator;

use anyhow::Result;
use std::path::Path;
use crate::config::PiceConfig;

/// Open the metrics database for a project.
/// Returns None if the DB doesn't exist (project not initialized).
pub fn open_metrics_db(project_root: &Path) -> Result<Option<db::MetricsDb>> {
    let config_path = project_root.join(".pice/config.toml");
    let config = PiceConfig::load(&config_path).unwrap_or_else(|_| PiceConfig::default());
    let db_path = project_root.join(&config.metrics.db_path);
    if !db_path.exists() {
        return Ok(None);
    }
    Ok(Some(db::MetricsDb::open(&db_path)?))
}
```

### Task 14: Create Telemetry Module

**File:** `crates/pice-cli/src/metrics/telemetry.rs`
**Action:** CREATE
**Description:** Telemetry is opt-in, anonymized, transparent, and non-blocking. This module handles:

1. **Anonymization**: Strip file paths, project names, user info from payloads. Keep only: evaluation scores, pass/fail, tier, provider type, event type, timestamp.
2. **JSONL logging**: Append every outbound payload to `.pice/telemetry-log.jsonl` before sending.
3. **Queue processing**: Read pending entries from `telemetry_queue`, send via HTTP POST, mark as sent.
4. **Silent failures**: HTTP errors logged at `debug` level only. Never surface to user.

```rust
pub struct TelemetryClient {
    endpoint: String,
    log_path: PathBuf,
}

impl TelemetryClient {
    pub fn new(config: &TelemetryConfig, project_root: &Path) -> Self
    
    /// Queue a telemetry event. Anonymizes the payload before storing.
    pub fn queue_event(&self, db: &MetricsDb, event: &TelemetryEvent) -> Result<()>
    
    /// Process the queue: send pending events, log to JSONL, mark sent.
    /// Called opportunistically (e.g., at CLI exit). Non-blocking.
    pub async fn flush(&self, db: &MetricsDb) -> Result<()>
    
    /// Read the JSONL log for `pice telemetry show` (future command).
    pub fn read_log(&self, limit: usize) -> Result<Vec<String>>
}

pub struct TelemetryEvent {
    pub event_type: String,    // "evaluation", "loop_complete", etc.
    pub tier: Option<u8>,
    pub passed: Option<bool>,
    pub score_avg: Option<f64>,
    pub provider_type: String, // "claude-code", "codex"
    pub timestamp: String,
}
```

For MVP: queue events locally and log to JSONL. HTTP sending can be stubbed (the endpoint `telemetry.pice.dev` doesn't exist yet). The queue and logging infrastructure must work.

## Contract

> This contract defines testable success criteria. It is negotiated during `/plan-feature` and graded by `/evaluate` after implementation. For Tier 2+, `/evaluate` runs both a Claude evaluator (formal contract grading) and a GPT-5.4 adversarial review (design challenge) in parallel. Both evaluators see ONLY this contract, the code diff, and CLAUDE.md — not the planning conversation.

### Tier

**2** — determines evaluation depth and model coverage:

| Tier | Claude Evaluator | Codex Adversarial (GPT-5.4 high) | Use When |
| ---- | ---------------- | -------------------------------- | -------- |

**Rules for writing criteria:**

- Each criterion must be independently testable — no "works well" or "looks good"

## Notes

- **No HTTP sending in MVP**: The telemetry endpoint (`telemetry.pice.dev`) doesn't exist. Phase 4 builds the queue + JSONL infrastructure. Actual HTTP sending is Phase 5 polish.
- **State machine deferred again**: `.pice/state.json` for formal state transitions was considered but deferred. The metrics DB + filesystem scanning remain sufficient. State tracking can be added in Phase 5 if metrics reveal a need.
- **`rusqlite` bundled feature**: Compiles SQLite from C source, adding ~5s to first build but eliminating system library dependency. Worth it for cross-platform reliability.
- **CSV export is raw data**: `--csv` exports per-evaluation rows, not the aggregate summary. This lets users import into spreadsheets for custom analysis.
- **`provider_name` on orchestrator**: Phase 2 scaffolded this field with `#[allow(dead_code)]`. Phase 4 removes the marker and uses it in metrics recording. However, the current instrumentation approach accesses provider names from config, not from the orchestrator. The dead code marker removal for `provider_name` should only happen if we actually read it — if not, leave it for Phase 5.
