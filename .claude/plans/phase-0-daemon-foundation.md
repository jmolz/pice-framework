# Feature: Phase 0 — Daemon Foundation Refactor

> Generated by `/plan-feature` — read CLAUDE.md, PRDv2.md, and the referenced files before implementing.

## Overview

Split the v0.1 monolithic `pice-cli` crate into three crates — `pice-cli` (thin CLI adapter), `pice-daemon` (long-running orchestration daemon), and `pice-core` (shared pure-logic library) — then wire them together so every v0.1 command executes by dispatching to the daemon over a Unix socket / named pipe. **This is a pure refactor**: no new user-visible features, no behavior changes. Every existing test must still pass.

This is the foundation for all of v0.2. Stack Loops, workflow YAML, seam checks, adaptive algorithms, approval gates, and background execution all assume this split exists. Until Phase 0 lands, v0.2 work has nowhere to live.

## User Story

As a PICE maintainer, I want the Rust core refactored into `pice-cli` + `pice-daemon` + `pice-core` crates with a working daemon RPC loop, so that I can add v0.2 features (Stack Loops, workflow YAML, background execution) without touching the binary boundary again.

## Success Criteria

- [ ] Workspace contains 4 crates: `pice-cli`, `pice-daemon` (new), `pice-core` (new), `pice-protocol` (unchanged)
- [ ] `pice-daemon` is both a library AND a binary (has `src/lib.rs` and `src/main.rs`)
- [ ] `pice-core` has zero async dependencies and zero network dependencies
- [ ] All 168 existing Rust tests pass without modification (modulo import path updates)
- [ ] All 49 existing TypeScript tests pass
- [ ] Every v0.1 command (`init`, `prime`, `plan`, `execute`, `evaluate`, `review`, `commit`, `handoff`, `status`, `metrics`, `benchmark`, `completions`) produces identical output to v0.1 — same stdout, same exit codes, same `--json` shape
- [ ] `pice` binary auto-starts `pice-daemon` on first use if not already running
- [ ] Daemon cold start (auto-start included) completes in < 500 ms on a warm cache
- [ ] Warm CLI → daemon → response round-trip for `pice --version` or `daemon/health` completes in < 50 ms
- [ ] `PICE_DAEMON_INLINE=1` bypasses the daemon and runs the orchestrator in-process (regression-safe escape hatch)
- [ ] `pice daemon {start|stop|status|restart|logs}` subcommands work
- [ ] Graceful shutdown: daemon responds to SIGTERM, flushes in-flight work, removes socket, exits within 10s
- [ ] Stale socket cleanup: a socket left over from an unclean shutdown is detected and removed on next daemon start
- [ ] Streaming output preservation: `pice plan`, `pice execute`, `pice review`, `pice prime` stream provider chunks to the terminal live, identical UX to v0.1
- [ ] Windows named pipe transport parity: daemon + CLI work on Windows with named pipes behind the same `DaemonTransport` trait
- [ ] All validation commands pass with zero errors (cargo fmt check, clippy -D warnings, cargo test, pnpm lint, pnpm typecheck, pnpm test, pnpm build, cargo build --release)
- [ ] No regressions in existing functionality — this is the single most important criterion

## Feature Metadata

- **Type**: Refactor
- **Complexity**: High
- **Systems Affected**: `crates/pice-cli/**` (split), `Cargo.toml` workspace, new `crates/pice-core/**`, new `crates/pice-daemon/**`, CI workflows (no changes expected, but watch for bin name changes), `npm/` distribution (ships a second binary)
- **New Dependencies**:
  - `thiserror` (already in `workspace.dependencies`, not currently used by `pice-cli`)
  - No truly new crate additions for Phase 0 — the daemon uses existing tokio features; Unix sockets are `tokio::net::UnixListener` / `UnixStream` (tokio already has `full` features)
  - Windows named pipes use `tokio::net::windows::named_pipe::*` (same `tokio` crate, no new deps)

```rust
// crates/pice-core/src/protocol/mod.rs
use serde::{Deserialize, Serialize};

pub mod methods;

#[derive(Debug, Serialize, Deserialize)]
pub struct DaemonRequest {
    pub jsonrpc: String,           // "2.0"
    pub id: u64,
    pub method: String,
    pub auth: String,              // bearer token (hex)
    #[serde(default)]
    pub params: serde_json::Value,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct DaemonResponse {
    pub jsonrpc: String,
    pub id: u64,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub result: Option<serde_json::Value>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub error: Option<DaemonError>,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct DaemonNotification {
    pub jsonrpc: String,
    pub method: String,
    pub params: serde_json::Value,
    // No id — notifications are fire-and-forget
}

#[derive(Debug, Serialize, Deserialize, thiserror::Error)]
#[error("daemon error {code}: {message}")]
pub struct DaemonError {
    pub code: i32,
    pub message: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub data: Option<serde_json::Value>,
}
```

## Notes

**Why one polymorphic `cli/dispatch` instead of per-command RPC methods?** Phase 0 is a refactor, not a new feature. The simplest daemon RPC surface that preserves behavior is one that mirrors the clap `Commands` enum. Per-command methods (`execute/create`, `evaluate/create`, etc.) will be added in v0.2 Phase 1+ when we need finer-grained control over layer-scoped execution. Adding them in Phase 0 would violate the "no new features" rule.

**Why does `pice-cli` depend on `pice-daemon` as a library?** Inline mode (`PICE_DAEMON_INLINE=1`) needs to run the orchestrator in-process without serializing through a socket. The cleanest way is for `pice-cli` to import `pice_daemon::inline::run_command` directly. The dependency chain is linear (`pice-cli → pice-daemon → pice-core → pice-protocol`), no cycles. Alternative considered: putting the orchestrator in `pice-core` with a feature flag for async — rejected because `pice-core` must stay pure.

**Why keep `aggregator.rs` in `pice-cli` instead of `pice-daemon`?** The `pice metrics` and `pice benchmark` commands do READ-ONLY queries against the SQLite DB. Having the CLI query the DB directly (instead of via daemon RPC) avoids a socket round-trip for reporting. The daemon owns writes; the CLI owns reads. Both import the schema types from `pice-daemon::metrics::db` as a shared library. This is a pragmatic choice — technically it violates the "all state goes through the daemon" principle, but only for read-only aggregation, and only until the dashboard lands in v0.3 (at which point the dashboard will do the same thing).

**What about Windows path handling?** The default socket path on Windows is `\\.\pipe\pice-daemon`, not a filesystem path. Unlike Unix sockets, named pipes don't have a filesystem representation that can be checked for permissions. Access control is via Windows ACL, and the default `ServerOptions::default()` creates pipes with owner-only access. The stale pipe detection is also different: pipes are cleaned up when the daemon exits, but an orphaned pipe from an unexpected kill may linger. The detection strategy is the same as Unix — try to connect; if it fails, consider it stale.

**Risk: Cargo workspace rebuild time.** Adding two new crates with significant shared code may increase initial compile time by 30–60%. Mitigation: `cargo check` performance is still fast; release builds are CI-only. If this becomes painful, revisit the crate split in a future phase.

**Risk: Daemon leaks across tests.** Integration tests that spawn real daemons must clean up the daemon process on test teardown. Use `tokio::process::Child` with explicit `kill()` in the test's drop or use `TestDaemon` fixture that wraps spawn+cleanup. Test runs in CI must not leave zombie daemon processes.

**Open question: should `pice-cli` log to the daemon's log file or its own stream?** For Phase 0, keep v0.1 behavior: CLI logs to stderr, daemon logs to `~/.pice/logs/daemon.log`. Do not unify. A future phase can add "view daemon logs via CLI" (`pice daemon logs --follow`), which is in scope for this phase (Task 24).

**Deferred to future phases (explicitly not Phase 0):**
- Manifest schema — not needed until Phase 1 introduces per-layer state
- `.pice/layers.toml`, `.pice/workflow.yaml` — not needed until Phase 2
- Seam checks — not needed until Phase 3
- Adaptive algorithms — not needed until Phase 4
- Worktrees — not needed until Phase 5
- Review gates — not needed until Phase 6
- Background mode — not needed until Phase 7 (Phase 0 keeps commands synchronous; the daemon runs them and returns when done, just like v0.1)
- Dashboard — v0.3

**Confidence for one-pass execution**: 6/10. This is a large refactor with many moving parts. The highest-risk tasks are 12 (orchestrator move), 19 (11 handlers), 21 (lifecycle + signal handling), and 22 (auto-start + streaming relay). I expect at least 2–3 iterations during execution. The comprehensive task list and strict test parity requirements should catch regressions early.
