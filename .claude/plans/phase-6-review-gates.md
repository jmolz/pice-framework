# Feature: PRDv2 Phase 6 — Human-in-the-Loop Review Gates (Final)

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.
> **Tier 3 plan. Two adversarial refinement cycles complete.** This revision applies 15 architectural fixes from Cycle 2 findings; remaining debatable items appear in `## Adversarial Review` as Acknowledged with reasoning.

## Overview

Add review-gate pause points to the Stack Loops orchestrator at **cohort boundaries**, without introducing new lock architecture or a new resume command. Key invariants:

- **Evaluate releases per-manifest locks between cohorts**, so gate decisions (which acquire the same locks) never deadlock against in-flight evaluation.
- **`pice evaluate plan.md` auto-detects existing non-terminal manifests** and resumes from the next cohort. No `--resume` flag.
- **CI / non-TTY runs exit 3** (`ExitJsonStatus::ReviewGatePending`) on gate fire, so scripts can loop `while [ $? -eq 3 ]; do pice review-gate … ; pice evaluate plan.md; done`.
- **TTY runs drive an in-process loop** via an injected `DecisionSource` trait: prompt (stderr) → `review-gate` RPC → re-invoke `evaluate` until terminal.
- **Timeouts enforced by a daemon-startup reconciler** against an injected `Clock` abstraction, so tests can use `tokio::time::pause()` faithfully.
- **Gate decisions use audit-before-manifest ordering**: SQLite write first (UNIQUE on `gate_id` gives CAS); manifest write second under the **same** tokio+fs2 lock the orchestrator uses.

## User Story

As a **reviewer on a team running PICE**, I want to **insert a mandatory human-review pause on infrastructure/deployment/low-confidence layers that survives tmux detach, daemon restart, CI job lifecycle, and concurrent reviewers**, so that **high-blast-radius changes are sanity-checked without coupling the orchestrator to my terminal, and every decision is persisted to a tamper-evident audit trail**.

## Success Criteria

- [ ] Every `workflow.review.trigger` grammar operator evaluates correctly against real layer contexts.
- [ ] Evaluate handler drains the current cohort on gate fire, persists `overall_status = pending-review`, and **returns from the handler** (orchestrator releases all locks).
- [ ] TTY-attached `pice evaluate` drives the prompt loop via `DecisionSource`; approve/skip/reject-with-retry re-invoke `pice evaluate plan.md` inline until terminal.
- [ ] Non-TTY / `--json` runs emit pending-gates summary and exit **3** (`review-gate-pending`).
- [ ] `pice review-gate --list` lists every pending gate; `pice review-gate --gate-id ID --decision approve|reject|skip [--reason TXT]` applies decisions keyed by `gate_id`.
- [ ] Approved/skipped gates release the feature; next `pice evaluate plan.md` picks up at the next cohort.
- [ ] Rejected gates with `reject_attempts_remaining > 0` decrement the counter and return the layer to `Pending`; the counter persists across re-gate events in the same feature run.
- [ ] Rejected gates with zero retries halt with `halted_by = gate_rejected` and exit 2 (`review-gate-rejected`).
- [ ] Skip keeps `LayerStatus::Passed`; only the gate's status becomes `Skipped` and an audit row records the skip.
- [ ] Timeouts fire via daemon-startup reconciler against the injected `Clock`; `gate/decide` prelude also applies timeouts in the same critical section.
- [ ] Audit row is written **before** the manifest mutation in `gate/decide`; manifest-write failure leaves an audit row but rolls back nothing visible (append-only).
- [ ] Schema migration `0.2 → 0.3` is soft: v0.2 manifests (pre-Phase-6) load successfully with an empty `gates: []` default; first save upgrades to 0.3.
- [ ] `pice status` shows `⏸ pending review: {layer}` on features with open gates.
- [ ] `pice audit gates [--feature F] [--since ISO8601] [--csv|--json]` exports the audit trail.
- [ ] `PICE_STATE_DIR` is honored by both the reconciler and `gate/list`.
- [ ] All validation commands pass with zero errors; `cargo test --features pty-tests` exercises the PTY prompt harness.
- [ ] No regression in Phase 5 cohort parallelism, cancellation, or existing evaluate lock semantics.

### Patterns to Follow

**Halt prefix registry (flat underscore, matching `sprt_*`):**

```rust
// pice-core/src/gate.rs
pub const HALTED_GATE_REJECTED: &str = "gate_rejected";
pub const HALTED_GATE_TIMEOUT_REJECT: &str = "gate_timeout_reject";
// approve/skip are NOT halts — they appear only in the audit.decision column, not halted_by

pub fn is_gate_halt(halted_by: &str) -> bool {
    matches!(halted_by, HALTED_GATE_REJECTED | HALTED_GATE_TIMEOUT_REJECT)
}
```

Only `reject` and `timeout_reject` produce `halted_by` values. `approve`/`timeout_approve`/`skip`/`timeout_skip` affect the gate and audit only.

**Clock abstraction:**

```rust
// pice-core/src/clock.rs
#[async_trait]
pub trait Clock: Send + Sync {
    fn now(&self) -> DateTime<Utc>;
    async fn sleep_until(&self, at: DateTime<Utc>);
}
pub struct SystemClock;
pub struct MockClock { /* shared Arc<Mutex<DateTime<Utc>>> + wake Notify */ }
```

`DaemonContext::clock: Arc<dyn Clock>` defaults to `Arc::new(SystemClock)`. Tests inject `MockClock`.

**Lock model (UNCHANGED from today + explicit release points):**

- `handlers/evaluate.rs` holds the per-manifest tokio mutex + fs2 file lock **only during manifest mutations**, not across an entire `run_stack_loops` call. Task 5 introduces the explicit acquire→mutate→release pattern at cohort boundaries.
- `ReviewGate::Decide` handler acquires the same per-manifest locks inside its critical section, performs audit write + manifest mutation, releases. No new mutator abstraction.

### Task 5: Orchestrator — release locks between cohorts + cohort-boundary gate check

**File:** `crates/pice-daemon/src/handlers/evaluate.rs`, `crates/pice-daemon/src/orchestrator/stack_loops.rs`
**Action:** MODIFY
**Description:**

**Lock release pattern** (the single biggest architectural change — resolves Codex Cycle-2 C2 self-deadlock):

```rust
// PSEUDOCODE — real code preserves existing atomic-save helpers.
// In handlers::evaluate::run:
for cohort_index in 0..n_cohorts {
    // 1. Acquire locks, mark cohort in-progress, save, release.
    { let _g = acquire_manifest_locks(feature_id).await?;
      manifest.cohorts[cohort_index].status = InProgress;
      save_manifest_atomic(&manifest)?; }            // locks drop here
    // 2. Run cohort WITHOUT holding locks. Providers run in parallel JoinSet.
    let results = run_cohort_parallel(cohort_index, …).await;
    // 3. Re-acquire locks, write results + any gates, release, return-or-continue.
    let _g = acquire_manifest_locks(feature_id).await?;
    let manifest = load_manifest_fresh(feature_id)?;  // stale-read protection
    apply_cohort_results(&mut manifest, &results);
    let new_gates = check_gates_for_cohort(&manifest, &workflow, &ctx.clock, cohort_index);
    if !new_gates.is_empty() {
        manifest.gates.extend(new_gates);
        manifest.overall_status = PendingReview;
        save_manifest_atomic(&manifest)?;
        ctx.gate_wake.notify_one();                   // wake reconciler
        return Ok(pending_review_response(&manifest));
    }
    save_manifest_atomic(&manifest)?;
    // locks drop; next cohort iterates
}
```

`check_gates_for_cohort` (pure fn in `stack_loops.rs`):
- For each layer whose cohort-index matches and whose status transitioned to `Passed`:
  1. Build `TriggerContext::for_layer(…)`.
  2. Resolve effective trigger: `LayerOverride.require_review == Some(true)` → literal `true`; else `LayerOverride.trigger`; else `workflow.review.trigger` (if `enabled`).
  3. Parse+evaluate. If true:
     - **Reject-counter persistence rule**: if this layer already has a prior `GateEntry` in `manifest.gates` (from a previous gate cycle), the new gate inherits `reject_attempts_remaining` from the prior gate (not reset). Otherwise initialize from effective `retry_on_reject`. This implements "counter persists across re-gates within one feature run" (Codex Cycle-2 C6). The counter is explicitly a **reject budget** — `approve` and `skip` do not consume it; `approve` is terminal regardless of remaining count.
     - Create `GateEntry` with pinned `timeout_at`, `on_timeout_action`, `reject_attempts_remaining`.
     - Set `layer.status = PendingReview`.

**Resume detection** (replaces `--resume` flag): in `handlers::evaluate::run`, before the cohort loop, load the existing manifest if present. If `overall_status == InProgress` (possible after reject-retry or approve+continuation), skip all layers with terminal status (Passed/Failed/Skipped) and start the cohort loop at the first non-terminal cohort. If `overall_status == PendingReview`, return immediately with `ReviewGatePending` — user must action gates first.

### Task 10: `ReviewGate` handler

**File:** `crates/pice-daemon/src/handlers/review_gate.rs` (new), `crates/pice-daemon/src/handlers/mod.rs`
**Action:** CREATE + MODIFY
**Description:**

`List`: read-only. Walks `state_dir` (`PICE_STATE_DIR`-aware), loads each manifest optimistically (no lock; retry once on torn-read deserialization failure), filters `status == Pending` gates. Returns `GateListResponse { gates }`.

`Decide`: inside the same per-manifest tokio+fs2 locks used by `evaluate`. Steps:

```
1. Acquire per-manifest locks via the existing helper used by evaluate.
2. Load manifest fresh.
3. Find gate by `gate_id`. Not found → release, return ReviewGateConflict.
4. Apply timeout prelude: let outcome = apply_timeout_if_expired(&mut gate, gate.on_timeout_action, ctx.clock.now()). If Some(outcome):
   a. write audit FIRST (audit_decision_string = outcome.audit_decision_string; reviewer = "timeout"; reason = None)
   b. write manifest mutations (gate.status, halted_by etc.)
   c. save manifest
   d. wake reconciler
   e. return the timeout outcome (not the requested decision)
5. Else apply manual decision:
   a. Compute decided_at = ctx.clock.now(); elapsed_seconds = decided_at - gate.requested_at
   b. Pre-compute manifest changes WITHOUT writing:
      - Approve → layer.status = Passed (already Passed; no-op), gate.status = Approved, gate.decision = "approve"
      - Skip → layer.status = Passed (UNCHANGED — critical semantic fix), gate.status = Skipped, gate.decision = "skip"
      - Reject with reject_attempts_remaining > 0 → decrement; layer.status = Pending; gate.status = Rejected; gate.decision = "reject"; overall_status unchanged (remains InProgress after gate decision clears).
      - Reject with reject_attempts_remaining == 0 → layer.status = Failed; gate.status = Rejected; gate.decision = "reject"; overall_status = Failed; halted_by = HALTED_GATE_REJECTED.
   c. **Write audit row FIRST** via insert_gate_decision. If fails, release locks and return ExitJsonStatus::MetricsPersistFailed. (Audit-before-manifest ordering — closes Codex Cycle-2 C8.)
   d. Write manifest: apply the pre-computed changes + clear PendingReview from overall_status if no remaining Pending gates.
   e. save manifest.
   f. wake reconciler (for the next earliest_timeout recompute).
6. Release locks. Return GateDecideResponse with updated `pending_gates`.
```

`reviewer` is a field on the RPC — resolved CLI-side (`$USER` or fallback). Daemon never reads env. `elapsed_seconds` is computed daemon-side.

## Notes

**Architecture summary.** Phase 6 halts the orchestrator at cohort boundaries (not layer boundaries) and releases per-manifest locks between cohorts so the gate RPC handler can acquire them without nested re-entry. The CLI drives resumption: in TTY mode, it loops prompt → decide → re-invoke-evaluate until terminal; in CI/JSON mode, it exits 3 and lets a shell loop drive the same cycle. Timeouts are enforced by an internal daemon-startup `GateReconciler` using an injected `Clock` abstraction — `tokio::test` code can pause time and advance deterministically. SQLite `gate_decisions` has UNIQUE on `gate_id` which gives CAS for free; audit is written before manifest to prevent "decided without audit" states.

**Why drop first-class RPC methods.** The 11 existing handlers all route through `CommandRequest`; adding two first-class methods would have required building client transport helpers that don't exist. The gate handler is 80% identical in structure to any other `handlers/*.rs`, so routing it through `CommandRequest::ReviewGate` saves ~150 lines of adapter and transport code. Phase 7 dashboard work can promote to first-class if genuine demand appears.

**Why auto-detect resume instead of `--resume` flag.** The daemon already has to load the manifest before doing anything; detecting `overall_status == InProgress` with layers already Passed is trivial. Removing the flag removes a user-visible surface that Phase 7 would have to re-spec anyway. Matches `pice status` behavior (also auto-detects).

**Why Clock trait with a default SystemClock.** The reconciler must test timeout behavior deterministically. `tokio::time::pause()` only pauses timers — `Utc::now()` keeps advancing. An injected clock lets both the reconciler scheduler AND the `resolve_timeout_action` pure fn share the same time source in tests.
