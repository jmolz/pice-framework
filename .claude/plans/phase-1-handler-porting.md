# Feature: Phase 1 — Port Daemon Handler Stubs to Real Logic

> Generated by `/plan-feature` — read CLAUDE.md, PRDv2.md, and `.claude/rules/daemon.md` before implementing.

## Overview

Port all 11 daemon handler stubs from Phase 0 placeholders to real implementations that replicate v0.1 behavior through the new daemon architecture. This is the bridge between Phase 0 (structural refactor) and PRDv2 Phase 1 (layer detection): every v0.2 feature depends on the daemon handlers actually working.

## User Story

As a PICE user on v0.2, I want every `pice` command to produce the same output as v0.1, so that the daemon refactor is invisible and I can continue using the PICE workflow.

## Success Criteria

- [ ] All 22 currently-ignored integration tests in `command_integration.rs` pass
- [ ] All 11 `pice` commands produce output identical to v0.1 behavior (same stdout, same exit codes, same `--json` shape)
- [ ] Both inline mode (`PICE_DAEMON_INLINE=1`) and socket mode produce identical results
- [ ] Streaming commands (`prime`, `plan`, `execute`, `review`) stream provider output live to the terminal
- [ ] `pice evaluate` runs dual-model adversarial evaluation and records results to metrics DB
- [ ] `pice commit` generates commit messages from staged diffs and executes `git commit`
- [ ] `pice status` parses plan files, queries metrics DB, and renders a table with evaluation columns
- [ ] `pice metrics` aggregates evaluation data from SQLite and formats as table/JSON/CSV
- [ ] All validation commands pass with zero errors
- [ ] No regressions in the 11 existing help tests or the 242 Rust tests currently passing

## Feature Metadata

- **Type**: Refactor (porting logic from v0.1 command files into daemon handler signatures)
- **Complexity**: High (11 handlers, each with distinct logic; streaming, dual-model, git operations)
- **Systems Affected**: `crates/pice-daemon/src/handlers/*.rs` (primary), `crates/pice-daemon/src/orchestrator/` (session lifecycle), `crates/pice-daemon/src/prompt/` (prompt builders), `crates/pice-daemon/src/metrics/` (recording), `crates/pice-cli/tests/command_integration.rs` (un-ignore)

### Patterns to Follow

**Handler signature (all 11 handlers):**

```rust
pub async fn run(
    req: XxxRequest,
    ctx: &DaemonContext,
    sink: &dyn StreamSink,
) -> Result<CommandResponse>
```

**JSON vs Text response pattern:**

```rust
if req.json {
    Ok(CommandResponse::Json { value: serde_json::json!({...}) })
} else {
    Ok(CommandResponse::Text { content: formatted_output })
}
```

**Streaming session pattern (prime, plan, execute, review):**

```rust
let config = pice_core::config::load_config(&project_root)?;
let mut orchestrator = ProviderOrchestrator::start("claude-code", &config).await?;
orchestrator.on_notification(streaming_handler(sink));
let result = run_session(&mut orchestrator, &project_root, &prompt).await;
orchestrator.shutdown(Duration::from_secs(10)).await;
result?;
```

**Capture session pattern (commit, handoff):**

```rust
let mut orchestrator = ProviderOrchestrator::start("claude-code", &config).await?;
let captured = run_session_and_capture(&mut orchestrator, &project_root, &prompt, sink).await;
orchestrator.shutdown(Duration::from_secs(10)).await;
let text = captured?;
```

**Non-fatal metrics recording:**

```rust
if let Ok(Some(db)) = metrics::open_metrics_db(&project_root) {
    if let Err(e) = store::record_evaluation(&db, ...) {
        tracing::warn!("metrics recording failed: {e}");
    }
}
```

**Error handling:**

```rust
// File-not-found → CommandResponse::Exit
if !plan_path.exists() {
    return Ok(CommandResponse::Exit {
        code: 1,
        message: format!("plan file not found: {}", plan_path.display()),
    });
}
```

**Imports:**

```rust

use crate::orchestrator::{ProviderOrchestrator, session, stream::StreamSink};
```

## Notes

### Key Decisions Needed During Implementation

1. **Init handler and embedded templates**: Templates are `rust-embed`'d in `pice-cli`, not `pice-daemon`. Options: (a) move embedding to `pice-core`, (b) keep init as a CLI-local operation that delegates only the metrics DB creation to the daemon, (c) embed in both crates. Option (b) is simplest and maintains the crate boundary (templates are a CLI concern).

2. **DaemonContext extension**: The current `DaemonContext` has no `project_root` or `config`. These are needed by every handler. Adding them is Task 1-2.

3. **Socket-mode streaming**: The router's `handle_dispatch()` currently passes `NullSink` to handlers. For socket mode to stream, this needs a `SocketSink` that sends `cli/stream-chunk` JSON-RPC notifications back to the CLI. This is a separate concern from handler porting — inline mode (which the tests use) already has `TerminalSink`. Socket-mode streaming can be a follow-up task or part of this phase.

### Risks

- **Provider availability in CI**: Integration tests that use the stub provider need `pnpm build` first (the CI fix we just shipped addresses this).
- **Metrics DB path**: Different platforms have different home directories. Tests should use temp dirs, not `~/.pice/`.
- **Git operations in tests**: `commit` handler tests need a real git repo with staged changes. The test fixtures need careful setup/teardown.
