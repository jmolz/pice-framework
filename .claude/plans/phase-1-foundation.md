# Feature: Phase 1 ‚Äî Foundation

> Generated by `/plan-feature` ‚Äî read CLAUDE.md and referenced files before implementing.

## Overview

Set up the PICE CLI monorepo from scratch: Rust workspace with two crates (`pice-cli` binary and `pice-protocol` library), TypeScript workspace with three packages (`provider-protocol`, `provider-base`, stub provider), the provider host that spawns and communicates with providers via JSON-RPC over stdio, the `pice init` scaffolding command, and CLI skeletons for all 11 commands.

## User Story

As a developer starting a new project, I want to run `pice init` so that my project is immediately set up with the PICE framework (.claude/ directory with commands, templates, hooks, docs, rules, and .pice/ directory with config).

## Success Criteria

- [ ] `cargo build` produces a `pice` binary that starts in under 100ms
- [ ] `pice --help` shows all 11 commands with descriptions
- [ ] `pice init` scaffolds correct `.claude/` and `.pice/` directories from embedded templates
- [ ] `pice init` is idempotent ‚Äî running twice doesn't overwrite existing files

- [ ] `pice-protocol` crate and `@pice/provider-protocol` package define identical message types

- [ ] `cargo test` passes with zero warnings
- [ ] `pnpm build && pnpm test` passes for all TS packages

- [ ] No regressions in existing functionality (greenfield ‚Äî baseline is clean)

### Task 4: @pice/provider-base (TypeScript)

**Files:** `packages/provider-base/package.json`, `packages/provider-base/tsconfig.json`, `packages/provider-base/src/index.ts`, `packages/provider-base/src/transport.ts`, `packages/provider-base/src/provider.ts`, `packages/provider-base/src/__tests__/transport.test.ts`, `packages/provider-base/src/__tests__/provider.test.ts`
**Action:** CREATE
**Description:** Shared provider infrastructure. `StdioTransport` reads newline-delimited JSON-RPC from stdin, dispatches to method handlers, writes responses to stdout, logs to stderr. `BaseProvider` abstract class handles initialize/shutdown lifecycle, capability declaration, and method dispatch. Tests verify transport parsing and provider lifecycle.
**Depends on:** Task 3
**Validate:** `cd packages/provider-base && pnpm test`

**StdioTransport key behaviors:**
- Read stdin line-by-line, parse each line as JSON-RPC
- Route requests to registered method handlers
- Write JSON-RPC responses to stdout (one per line)
- All logging via `process.stderr.write()` ‚Äî NEVER stdout
- Handle parse errors gracefully (send JSON-RPC error, don't crash)

**BaseProvider key behaviors:**
- Constructor registers core methods: `initialize`, `shutdown`, `capabilities`
- `initialize` handler stores config, sets `isInitialized` flag, returns capabilities
- `shutdown` handler cleans up and exits
- Abstract `dispatch` method for provider-specific methods
- Guard: reject non-initialize methods if not initialized

## Notes

- **Phase 1 is foundation only.** No AI SDK integration, no actual orchestration, no metrics collection. Those come in Phases 2-4.
- **The stub provider is critical** ‚Äî it enables testing the provider protocol without any API keys or AI service dependencies. All CI tests should use the stub.
- **Template files are a snapshot** ‚Äî they're copied from the current `.claude/` directory. Future updates to the PICE methodology will need to update `templates/` and rebuild.
- **The JSON-RPC protocol is the most important thing to get right** ‚Äî it's the contract between Rust and TypeScript, and between the core and all future providers. Invest in thorough serialization tests.
- **Potential follow-up:** If `@anthropic-ai/claude-code` SDK has limitations discovered during Phase 2, the provider architecture allows falling back to CLI subprocess mode without changing the core.
