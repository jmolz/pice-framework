## Overview

Implement `.pice/workflow.yaml` ‚Äî the committable, hierarchically-overridable pipeline definition ‚Äî and the `pice validate` command that catches schema errors, floor violations, trigger-expression syntax errors, and cross-reference errors before evaluation runs. Also ship 5 reference presets so users can start from curated baselines. This is PRDv2 Feature 4 in full.

## User Story

As a team lead, I want to commit `.pice/workflow.yaml` to the repo so every team member runs the same verification pipeline with the same tier thresholds, budgets, and review gates ‚Äî while still allowing each developer to RESTRICT (but never relax) those settings in `~/.pice/workflow.yaml`. And I want `pice validate` to fail fast on misconfiguration before wasting provider tokens on a bad run.

## Notes

**Why this plan excludes provider wiring.** Per the user's directive "PRD phases = plans," Phase 2 is Workflow YAML per PRDv2 lines 2046‚Äì2058. The provider-wiring work the previous HANDOFF called "Phase 2" is actually unfinished Phase 1 scope. See the "Prerequisite Gap" section at the top of this document.

**Why layer override plumbing is in scope even though full effects depend on future phases.** PRDv2 Phase 2 explicitly lists "Layer override plumbing into Stack Loops orchestrator" as in-scope. "Plumbing" means the code path exists; effects can be fully exercised once Phase 1 completion + Phase 4 adaptive + Phase 6 gates land. Task 16 delivers observable plumbing: the manifest records the effective tier per layer, so changes to workflow.yaml are visible even without the downstream machinery.

**Why floor-based merge is threshold-10.** Team leads commit workflow.yaml to enforce standards (tier floors, budget ceilings, review requirements). If a developer's user workflow could relax those floors, the whole point of the feature is defeated. This is the same security posture as a .editorconfig or a CODEOWNERS file ‚Äî individual overrides that bypass team policies are unacceptable.

**Why a hand-written trigger parser rather than `nom` or `pest`.** The grammar is tiny (~6 identifiers, ~6 operators, 3 literals). Recursive-descent by hand is well under 200 lines, has zero new dependencies, and produces better line/column diagnostics than nom's combinator error traces. `pest` would require a grammar file to ship. Not worth it.

**Why `BTreeMap` over `HashMap`.** Deterministic serialization order: YAML roundtrips, error messages, and diffs all benefit. The performance cost is irrelevant at this scale (< 50 layers, < 20 triggers).

**Future work unblocked by Phase 2.**
- Phase 3: Seam Verification reads the `seams` section of workflow.yaml
- Phase 4: Adaptive Evaluation reads `evaluate.adaptive_algorithm` and `defaults.min_confidence`
- Phase 6: Review Gates reads `review.trigger` and uses the same trigger evaluator
- Phase 7: Background Execution honors `defaults.max_parallelism`
- Phase 8: `pice init --preset strict` becomes a one-line addition

**Known limitations accepted for this phase.**
- No `pice init --preset NAME` ‚Äî presets are reference files only, copy-paste by user
- Cost cap behavior (`halt`/`warn`/`continue`) parses but is enforced in Phase 4 (no cost events yet)
- `adaptive_algorithm` parses but has no effect until Phase 4
- `seams` section parses but checks don't run until Phase 3
- `review` section parses but gates don't fire until Phase 6
- Layer override `require_review`, `max_passes`, `min_confidence`, `budget_usd` are parsed and threaded but have no runtime effect until adjacent phases land

These limitations are accepted: Phase 2 is the plumbing; other phases turn on the water.
