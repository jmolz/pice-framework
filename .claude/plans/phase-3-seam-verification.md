# Feature: PRDv2 Phase 3 — Seam Verification

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Wire the already-parsed `seams` section of `workflow.yaml` + `layers.toml` into Stack Loops as a first-class verification pass. Introduces `pice-core::seam` (trait + context + result + registry), a default library of 12 static checks (one per PRDv2 failure category), SQLite schema migration for `seam_findings`, provider-protocol extension (`evaluate/create.seamChecks`), and orchestrator wiring that runs checks after per-layer contract grading — populating the already-present `LayerResult.seam_checks` field. PRDv2 Feature 6.

## User Story

As a developer running `pice evaluate` on a multi-layer change, I want seam checks to run automatically at every declared layer boundary so that I catch configuration/schema/protocol mismatches — the 68% of outage causes that v0.1 contracts miss — before the feature ships.

## Success Criteria

- [ ] `pice-core::seam` module exists with `SeamCheck` trait, `SeamContext`, `SeamResult`, `Registry`, and `defaults` submodule
- [ ] Default registry contains at least one check per PRDv2 category (12 total), each tagged with `category: u8` 1–12
- [ ] Workflow validator rejects unknown seam check IDs and unknown boundary names (referencing `layers.order`)
- [ ] `workflow.yaml [seams]` **overrides** `layers.toml [seams]` per-boundary when present (explicit replacement, not merge — documented invariant)
- [ ] Stack Loops orchestrator runs checks after contract grading per layer and populates `LayerResult.seam_checks`
- [ ] Seam check failure marks layer status as `Failed` (never `Passed`) — fail-closed
- [ ] Manifest roundtrip test: `seam_checks` field serializes/deserializes with `category`, `boundary`, `status`, `details`
- [ ] SQLite migration v2 creates `seam_findings(id, evaluation_id, layer, boundary, check_id, category, status, details, created_at)` with FK to `evaluations`
- [ ] Provider protocol `evaluate/create` accepts optional `seamChecks: SeamCheckSpec[]` (Rust + TS both have roundtrip tests)
- [ ] Static checks are deterministic and complete in <100ms each on fixtures
- [ ] Context isolation: seam check context exposes only the layer's boundary diff + boundary-relevant files, NEVER other layers' contracts or findings
- [ ] E2E fixture tests pass for: env-var mismatch (category 1), ORM↔migration schema drift (category 9), OpenAPI↔handler response shape (category 3)
- [ ] `pice evaluate --json` reports seam findings in `layers[].seam_checks[]` and exits 2 when any finding is `Failed`
- [ ] All validation commands pass with zero errors; no regressions

### Task 2 — Workflow `seams` validation

- **Files (MODIFY):** `crates/pice-core/src/workflow/validate.rs`, `crates/pice-core/src/workflow/mod.rs` (export `validate_seams`)
- **New function:** `pub fn validate_seams(workflow: &WorkflowConfig, layers: &LayersConfig, registry: &Registry) -> Result<(), Vec<SeamValidationError>>`
- **Rules enforced (one error per violation, collected not fail-fast):**
  1. Boundary key parses as `"A↔B"` or `"A<->B"` (two non-empty layer tokens)
  2. `A` and `B` both appear in `layers.order`
  3. `A != B`
  4. Every check ID in the boundary's list exists in `registry.ids_in_order()`
  5. Same check ID cannot repeat within a single boundary's list
  6. No inverted duplicates: if `"A↔B"` exists, `"B↔A"` must not exist. Canonicalization: alphabetical `(A <= B)` only; the other form is rejected at validation (not silently merged)
- **Call sites:** invoked from the top-level `validate_workflow(...)`  in `workflow/mod.rs` alongside `validate_triggers`
- **Schema hardening:** add `#[serde(deny_unknown_fields)]` to any new seam-related wrapper structs
- **Tests:** 1 per rule above (6 rejection tests) + 1 happy-path + 1 stale-field-name rejection
- **Depends on:** Task 1
- **Validate:** `cargo test -p pice-core workflow::validate::tests::seam`

### Task 3 — Seam merge semantics (workflow per-boundary override)

- **Files (MODIFY):** `crates/pice-core/src/workflow/merge.rs` (+ tests)
- **New function:** `pub fn merge_seams(framework: &BTreeMap<...>, project: &BTreeMap<...>, user: Option<&BTreeMap<...>>) -> BTreeMap<String, Vec<String>>`
- **Rules (documented in a doc comment at the fn, same style as Phase 2 floor-merge docs):**
  - **framework → project:** overlay. Project may add/remove boundaries freely vs framework defaults (framework is baseline, not floor)
  - **project → user (workflow.yaml):** per-boundary replacement, with a FLOOR on boundary existence
    - If project declares boundary `"A↔B"`, user MUST declare it too or omit it (keep project list)
    - If user declares `"A↔B" = [...non-empty...]`, the list REPLACES the project list (user override wins)
    - If user declares `"A↔B" = []` (empty) for a project-declared boundary → validation error (cannot silently remove)
    - User may add NEW boundaries not declared by project (pure additions allowed)
- **Not floor-guarded:** direction of check-list growth isn't monotonic (adding a strict check vs swapping a strict for a lenient one is legitimate project-specific choice) — mirrors `max_passes` exemption from Phase 2
- **Tests (minimum):** replace-wins, omit-preserves-project, empty-list-rejects, user-adds-new-boundary, framework-overlay-not-floor, user-list-ordering-preserved
- **Depends on:** Task 2
- **Validate:** `cargo test -p pice-core workflow::merge::tests::seam`

### Task 8 — Seam runner (pure orchestration helper)

- **Files (CREATE):** `crates/pice-daemon/src/orchestrator/seam_runner.rs`
- **Files (MODIFY):** `crates/pice-daemon/src/orchestrator/mod.rs` (add `mod seam_runner; pub(crate) use seam_runner::run_seams_for_layer;`)
- **New function signature:**
  ```rust
  pub(crate) fn run_seams_for_layer(
      layer_name: &str,
      active_layers: &HashSet<String>,
      merged_seams: &BTreeMap<String, Vec<String>>,
      registry: &Registry,
      repo_root: &Path,
      full_diff: &str,
      layer_paths: &BTreeMap<String, Vec<PathBuf>>,
  ) -> Vec<SeamCheckResult>
  ```
- **Algorithm:**
  1. Iterate `merged_seams` entries; for each boundary `"A↔B"`, include it if `layer_name ∈ {A, B}` AND both `A` and `B` are in `active_layers`
  2. For each applicable boundary, resolve each check ID via `registry.get(&id)`; if missing, emit `SeamCheckResult { status: Failed, details: Some("check ID not registered: ..."), ... }` and continue
  3. Build `SeamContext` scoped to that boundary: `filtered_diff` = `full_diff` filtered to paths in `layer_paths[A] ∪ layer_paths[B]`; `boundary_files` = concrete file list from the same union
  4. Call `check.run(&ctx)` wrapped in a 100ms CPU timeout via `std::thread::spawn` + `std::sync::mpsc::Receiver::recv_timeout`. On timeout: join thread (detached if poisoned), emit `Warning` with message "seam check exceeded 100ms budget" (never panic)
  5. Convert `SeamResult` → `SeamCheckResult { name: check.id(), status, boundary, category: Some(check.category()), details }`
- **Context isolation guard:** `SeamContext` struct fields are ONLY the ones listed in Task 1; nothing else reachable via `Debug`, `Clone`, or public API
- **Tests:**
  - Single-layer, single-boundary: expected check runs once
  - Multi-boundary: only active boundaries evaluated

## Notes

**Phase 1 interaction.** Layers currently record `Pending` status with `model: "phase-1-pending"` because real provider evaluation isn't wired yet. Seam checks are **static and don't need a provider**, so Phase 3 ships independently of Phase 1 remediation. When a layer is `Pending` AND has a `Failed` seam finding, the layer status becomes `Failed` (seam findings override `Pending`). When Phase 1 remediation lands later, the ordering is: contract grade first → if Passed, run seams → seams can downgrade to Failed.

**Heuristic checks (categories 5, 10, 11).** Retry/cascade, cold-start ordering, and network-topology checks are intrinsically runtime-dependent for full fidelity. v0.2 ships static approximations that emit `Warning` findings — per PRDv2 line 1038, behavioral check stubs are expected placeholders for v0.4 implicit contract inference. Document this clearly in the check docstrings so users understand the v0.2 vs v0.4 split.

**Plugin crate discovery.** PRDv2 Feature 6 mentions `pice-seam-checks-{protocol}` plugin crates discovered at daemon startup. v0.2 scope: define the trait + registry so external crates CAN be authored; actual runtime plugin loading (dynamic library discovery) is deferred. Document the "build your own check" path; shipping an actual dynamic loader is a follow-up issue.

**`↔` vs `<->`.** Unicode `↔` reads well in docs but is awkward to type and search. Accept both `"A↔B"` and `"A<->B"` in parsing; canonicalize to `↔` for storage and error messages. Add a round-trip test.

**Adversarial review expectations.** Phase 2 took 5 Codex xhigh rounds to close silent bypasses. Expect similar scrutiny here. Likely attack surfaces for Codex to find:
- Seam check that returns `Passed` when it should return `Failed` (silent false-negative)
- `applies_to()` returning false too broadly → check never runs
- Context isolation leak via `Debug` derive printing whole-struct contents
- Merge semantics letting user empty-list a required boundary
- Exit-code path where Failed findings don't propagate to process exit 2
- SQLite insert happening even when evaluation is rolled back (orphan rows)

Prepare tests for each of these BEFORE the adversarial pass.

**Follow-ups (post v0.3 scope):**
- Dynamic plugin loading for community seam check crates (full `pice-seam-checks-grpc` pattern)
- LLM-backed checks for categories 5/10/11 (deferred to v0.4 implicit contract inference)
- `pice seam list` and `pice seam explain <id>` CLI subcommands (nice-to-have, easy to add after the registry exists)
- Arch-Experts-driven custom checks (v0.3)
